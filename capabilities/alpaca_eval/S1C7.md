## Capability Assessment

### ✗ Plan Pre-Specification Requirement
**Status:** Not Supported
**Evidence:** AlpacaEval lacks a formal analysis plan specification mechanism. Evaluation and analysis are conducted post-hoc:
- No analysis plan schema or registration system exists
- No pre-execution specification workflow is enforced
- Metrics are computed and reported without prior commitment to specific tests
- **Reference:** `/src/alpaca_eval/main.py` - evaluate() function accepts metric parameters but does not require pre-specification

**Gap:** Users can modify analysis parameters after seeing preliminary results, violating pre-registration principles.

---

### ✗ Sample Size Justification
**Status:** Not Supported
**Evidence:** No power analysis or sample size planning mechanisms are present:
- No power calculation functions available
- Sample size appears only post-hoc via `n_samples` parameter in analysis
- No a priori effect size assumptions documented
- No achieved power reporting after evaluation
- **Reference:** `/src/alpaca_eval/main.py` - n_samples parameter used retrospectively, not for planning

**Gap:** No mechanism to justify dataset size requirements before evaluation begins.

---

### ✗ Primary/Secondary Metric Specification
**Status:** Not Supported
**Evidence:** No formal hierarchy or designation of primary vs. exploratory metrics:
- Metrics are computed simultaneously without pre-specification of importance hierarchy
- Post-hoc selection via `sort_by` parameter in `/src/alpaca_eval/main.py`
- All metrics reported with equal weight regardless of prior designation
- No configuration schema for metric prioritization

**Gap:** Impossible to distinguish results that were planned vs. discovered during analysis.

---

### ✗ Significance Thresholds
**Status:** Hard-Coded Only (Not Configurable)
**Evidence:** Alpha threshold is fixed at 0.05 with no configuration mechanism:
- Hard-coded alpha value: 0.05 in `/src/alpaca_eval/plotting.py:727`
- Paired t-tests use this fixed threshold in `_pairwise_ttest()` (lines 816-827)
- No parameter to adjust alpha levels for different evaluation contexts
- **Code Reference:**
  ```python
  # /src/alpaca_eval/plotting.py:727
  alpha = 0.05  # Hard-coded
  ```

**Gap:** Cannot adjust significance thresholds for exploratory vs. confirmatory studies, or for different risk tolerance requirements.

---

### ✗ Multiple Comparison Corrections
**Status:** Not Implemented
**Evidence:** No correction methods for multiple testing are available:
- No Bonferroni correction implementation
- No Benjamini-Hochberg FDR (False Discovery Rate) method
- No Holm-Bonferroni step-down procedure
- No other sequential or adaptive correction methods
- Pairwise t-tests (`_pairwise_ttest()` at `/src/alpaca_eval/plotting.py:816-827`) do not apply corrections

**Gap:** When conducting multiple statistical tests (e.g., pairwise comparisons across models), Type I error is not controlled, inflating false positive rates.

---

### ✗ Plan Adherence Enforcement
**Status:** Not Supported

## Critical Gaps Summary

### Tier 1 (Must Have for Absent → Partial)
1. **Analysis Plan Schema** - Define structure for pre-specified analyses
2. **Pre-execution Specification** - Mechanism to register plans before evaluation
3. **Multiple Comparison Corrections** - Implement Bonferroni, FDR, or equivalent
4. **Configurable Alpha Levels** - Allow users to specify significance thresholds

### Tier 2 (Required for Partial → Present)
5. **Power Analysis** - Sample size calculation with effect size assumptions
6. **Primary/Secondary Metrics** - Formal designation and weighting system
7. **Plan Enforcement** - Audit trail and adherence checking
8. **Plan Versioning** - Track plan changes and document deviations

### Tier 3 (Optional, Beyond Present)
- Bootstrap confidence intervals with resampling specifications
- Sensitivity analysis framework
- Sequential decision rules and contingency planning
- Automated report generation following analysis plan

## Recommendations for Implementation

### Short-term (High Priority)
1. Introduce `AnalysisPlan` configuration class to specify metrics, tests, and thresholds before evaluation
2. Add Bonferroni correction option to `_pairwise_ttest()` function
3. Make alpha threshold a configurable parameter in plotting functions
4. Document statistical methodology and analysis decisions

### Medium-term (Medium Priority)
5. Implement power analysis module using `scipy.stats` or `statsmodels`
6. Add primary/secondary metric designation to evaluation configuration
7. Create plan versioning and audit logging system
8. Build analysis plan validator to prevent ad-hoc modifications

### Long-term (Nice to Have)
9. Implement bootstrap CI methods with customizable resampling parameters
10. Add sensitivity analysis framework for robustness checks
11. Create automated analysis report generator following pre-specified plan
12. Integrate with open science platforms (OSF, AsPredicted) for plan registration

---

## Conclusion

AlpacaEval currently operates at the **Absent (0-30% coverage)** level for analysis plan specification. While the harness demonstrates basic statistical capabilities (standard errors, stratified testing, multiple scoring rules), it completely lacks the infrastructure for pre-specifying analysis plans before evaluation execution.

To move to Partial compliance (30-80%), the framework needs:
1. A formal plan specification mechanism
2. Multiple comparison corrections
3. Configurable significance thresholds
4. Power analysis capabilities

To achieve Present compliance (80-100%), all eight capabilities must be implemented with enforcement mechanisms to prevent p-hacking and ensure research integrity.
