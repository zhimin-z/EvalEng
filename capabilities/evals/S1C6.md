[S1C6]: Cross-Validation Configuration
Overall Grade: Absent (0% coverage)

Features:
1. Data Split Configuration
   - Support: 0
   - Evidence: The harness uses a naming convention with splits (e.g., `eval_name.dev.v0`) as documented in docs/build-eval.md:59, but this is purely organizational. No harness-level mechanism exists to configure or enforce train/validation/test splits with customizable proportions. The split name is extracted from eval names (evals/cli/oaieval.py:192) but only used for metadata, not for actual data partitioning. Example benchmark files like hr_ml_agent_bench/benchmarks/house_price/env/train.py and feedback/env/train.py use sklearn's train_test_split, but these are user-provided task implementations, not harness features.
   - Documentation Quality: Minimal

2. Cross-Validation Method Selection
   - Support: 0
   - Evidence: No harness support for k-fold, stratified k-fold, leave-one-out, or time-series cross-validation methods. The only example found is in hr_ml_agent_bench/benchmarks/parkinsons_disease/env/train.py:149-157, which uses sklearn's GroupKFold(n_splits=8) and cross_val_score, but this is part of a user-provided benchmark task implementation, not a harness capability. The core evaluation framework in evals/eval.py focuses on running samples through completion functions without any cross-validation abstraction.
   - Documentation Quality: None

3. Data Leakage Prevention
   - Support: 0
   - Evidence: No enforcement mechanisms exist to prevent data leakage or ensure strict separation between train/validation/test sets. The harness processes samples independently through eval_sample (evals/eval.py:77-78) without validating set boundaries. The parkinsons_disease benchmark example uses GroupKFold to prevent patient-level leakage, but this is task-specific code, not a harness feature. No validation, warnings, or safeguards exist at the harness level.
   - Documentation Quality: None

4. Randomization Control
   - Support: 0
   - Evidence: While the harness provides seed parameters (--seed CLI flag with default 20220722 in evals/cli/oaieval.py:43, and SHUFFLE_SEED=123 in evals/eval.py:26), these control sample shuffling order and per-sample RNG initialization (evals/eval.py:135-136), not cross-validation fold generation or data split reproducibility. The seed mechanism ensures deterministic sample evaluation order but does not enable reproducible train/test splits as required for cross-validation. Example benchmarks set random seeds manually (e.g., feedback/env/train.py:17-21) outside the harness.
   - Documentation Quality: Minimal

5. Stratification Support
   - Support: 0
   - Evidence: No harness-level support for maintaining representative distributions of class labels or metadata attributes across splits. The harness treats all samples uniformly through _index_samples (evals/eval.py:30-38), which only shuffles and limits samples without stratification logic. The parkinsons_disease benchmark uses GroupKFold for group-based stratification, but this is implemented in the user's task code (hr_ml_agent_bench/benchmarks/parkinsons_disease/env/train.py:155), not provided by the harness.
   - Documentation Quality: None
