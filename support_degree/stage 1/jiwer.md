# JiWER - Stage 1 (CONFIGURE) Evaluation

## Summary
JiWER is a specialized library for computing ASR error metrics (WER, MER, CER, etc.) rather than a general evaluation framework. It lacks most configuration capabilities expected from an evaluation framework, as it operates as a metric calculation tool rather than an orchestration system. It has no dataset management, model configuration, environment automation, security features, or cost estimation capabilities.

## Detailed Analysis

### S1F1: Dataset Discovery and Logical Configuration (0/3)

Evidence of absence:
- `src/jiwer/measures.py` line 70-75:
```python
def wer(
    reference: Union[str, List[str]] = None,
    hypothesis: Union[str, List[str]] = None,
    reference_transform: Union[tr.Compose, tr.AbstractTransform] = wer_default,
    hypothesis_transform: Union[tr.Compose, tr.AbstractTransform] = wer_default,
) -> float:
```

Users pass raw strings directly - no dataset objects, no registration system, no metadata.

No schema definition: The library accepts any string/list of strings without validation or type constraints beyond basic Python typing.

No versioning: No dataset version tracking exists anywhere in the codebase.

No splits: Users manually partition their data before passing to JiWER.

Rating justification: 0 points - JiWER is not an evaluation framework but a metric computation library. It has no dataset abstraction layer.

### S1F2: Model and Backend Configuration (0/3)

Evidence of absence:
JiWER performs no model inference. It only compares text strings that have already been generated by models.

From `README.md`:
```python
from jiwer import wer

reference = "hello world"
hypothesis = "hello duck"

error = wer(reference, hypothesis)
```

The library assumes users have already run their ASR models and provides only the metric calculation layer.

No provider support: No OpenAI, Anthropic, HuggingFace, or any other provider integration exists.

Rating justification: 0 points - Not applicable to this type of library. JiWER is a post-processing tool, not an evaluation orchestrator.

### S1F3: Evaluation Parameters and Prompt Configuration (0/3)

Evidence of limited text transformation:
`src/jiwer/transformations.py` provides text preprocessing transforms:
```python
wer_default = tr.Compose([
    tr.RemoveMultipleSpaces(),
    tr.Strip(),
    tr.ReduceToListOfListOfWords(),
])
```

However, these are text normalization utilities, not prompt templates or parameter configurations.

No templating system: No Jinja2, variable substitution, or few-shot examples.

No metric configuration: Users select metrics by calling different functions (`wer()`, `cer()`, `mer()`), not via configuration.

Rating justification: 0 points - The library provides text transformations but no evaluation parameter configuration, prompting, or template systems.

### S1F4: Environment Setup and Dependency Management (2/3)

Strengths:
- Modern `pyproject.toml` with dependency groups:
```toml
[project]
requires-python = ">=3.8"
dependencies = [
    "click>=8.1.8",
    "rapidfuzz>=3.9.7",
]

[dependency-groups]
dev = ["ruff>=0.9.6", "pytest>=8.3.4", "pytest-benchmark>=4.0.0"]
docs = ["mkdocs>=1.6.1", ...]
```

- CI automation via GitHub Actions (`.github/workflows/pythonpackage.yml`):
```yaml
- name: Install uv
  uses: astral-sh/setup-uv@v5
  with:
    version: "0.5.26"

- name: Install dependencies
  run: uv sync --all-extras --all-groups
```

Weaknesses:
- No Docker image provided
- No Singularity support
- Manual setup required despite `uv` being available
- No hardware specification (GPU/CPU requirements)

Rating justification: 2 points - Good modern dependency management with pinned versions and CI automation, but lacks containerization and hardware specification.

### S1F5: Security and Access Control (0/3)

Evidence of absence:
Complete lack of security features throughout the codebase. This is expected for a metric computation library that doesn't handle credentials or external services.

No files contain authentication, authorization, encryption, or audit logging code.

Rating justification: 0 points - Not applicable; this is a pure computation library with no external service interaction.

### S1F6: Cost Estimation and Budget Planning (0/3)

Evidence of absence:
JiWER doesn't invoke any paid APIs or models. It only performs local computation on provided strings.

From `src/jiwer/alignment.py`, the library uses RapidFuzz for edit distance calculation:
```python
opcodes = rapidfuzz.distance.Levenshtein.opcodes(
    reference_sentence, hypothesis_sentence
)
```

All computation is local - no API calls, no token counting, no cost modeling.

Rating justification: 0 points - Not applicable; the library has no concept of API costs or resource budgeting.

## Overall Assessment

Total Score: 2/18 (11.1%)

JiWER is fundamentally misclassified as an "evaluation framework." It is a metric computation library designed to calculate error rates (WER, CER, MER, etc.) between reference and hypothesis text strings. It does not:

- Manage datasets
- Configure or invoke models
- Orchestrate evaluation runs
- Handle authentication or security
- Estimate costs

The library excels at what it's designed for: fast, accurate computation of ASR metrics with customizable text transformations. However, it lacks all configuration capabilities expected from a comprehensive evaluation framework as defined in the Stage 1 rubric.

### What JiWER Actually Provides:
1. Metric computation functions (`wer()`, `cer()`, `mer()`, etc.)
2. Text transformation utilities for normalization
3. Alignment visualization for error analysis
4. CLI tool for basic file-based metric calculation

### Appropriate Use Case:
JiWER should be used as a component within an evaluation framework, not as the framework itself. A proper evaluation framework would:
- Load datasets
- Run ASR models
- Pass results to JiWER for metric calculation
- Aggregate and report results

The only scored feature (S1F4) reflects the quality of its Python packaging, not evaluation framework capabilities.
