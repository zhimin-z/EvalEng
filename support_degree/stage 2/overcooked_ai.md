# Overcooked-AI - Stage 2 (PREPARE) Evaluation

## Summary

Overcooked-AI is a multi-agent reinforcement learning environment, not an evaluation framework. It provides a cooperative game environment for training and evaluating agents, but lacks dedicated dataset preparation, quality assessment, and evaluation infrastructure building capabilities. The repository focuses on game logic, agent training (PPO/BC - now deprecated), and human-AI interaction demos rather than evaluation artifact preparation.

## Detailed Analysis

### S2F1: Data Preprocessing and Physical Partitioning (1/3)

Evidence:

The repository contains basic data processing utilities in `src/human_aware_rl/human/process_dataframes.py`:

```python
def csv_to_df_pickle(layouts, data_dir, min_reward, out_data_path=MASTER_OUTPUT_PATH):
    """
    Loads, processes, and filters raw CSV data and saves as pickled DataFrame
    """
    # Basic loading and filtering
    df = pd.read_csv(os.path.join(data_dir, layout + '.csv'))
    df = format_trials_df(df)
    df = filter_trials(df, lambda trial: trial_reward(trial) >= min_reward)
```

However, this lacks:
- No caching mechanism: Data is loaded from scratch each time
- No versioning: No tracking of dataset versions or splits
- Limited preprocessing: Only trajectory data from game logs, no text/image/audio preprocessing
- No stratified splitting: Simple manual splitting without reproducible seeds

The README at `src/human_aware_rl/static/human_data/README.md` mentions "train" and "test" splits but these are pre-generated, not created by the framework:

```
cleaned/
  - Contains processed, filtered data as pickled pandas DataFrames
  - Data is divided into 2019 and 2020 experiments
  - Data is further divided into 'all', 'train', and 'test' sets
```

### S2F5: Model Artifact Validation (1/3)

Evidence:

Model loading has basic error handling but minimal validation:

```python
# From src/overcooked_demo/server/game.py
def get_policy(self, npc_id, idx=0):
    if npc_id.lower().startswith("rllib"):
        try:
            fpath = os.path.join(AGENT_DIR, npc_id, "agent")
            fix_bc_path(fpath)  # Path fixing, not validation
            agent = load_agent(fpath, agent_index=idx)
            return agent
        except Exception as e:
            raise IOError("Error loading Rllib Agent\n{}".format(e.__repr__()))
    else:
        try:
            fpath = os.path.join(AGENT_DIR, npc_id, "agent.pickle")
            with open(fpath, "rb") as f:
                return pickle.load(f)
        except Exception as e:
            raise IOError("Error loading agent\n{}".format(e.__repr__()))
```

Missing:
- No checksum validation: No SHA256 or cryptographic verification
- No version compatibility: No checks for model vs framework version
- No corruption detection: No integrity verification beyond load errors
- No config validation: Path fixing is manual, not schema-based

### S2F8: Data Contamination Detection (0/3)

Evidence:

While human gameplay data exists in `src/human_aware_rl/static/human_data/`, there are no contamination detection tools. The data processing code only performs basic filtering:

```python
# From src/human_aware_rl/human/process_dataframes.py
def filter_trials(df, predicate_func):
    """allows user to specify a filter function that filters entire trials"""
    good_trial_ids = []
    for trial_id, trial in df.groupby('trial_id'):
        if predicate_func(trial):
            good_trial_ids.append(trial_id)
    return df[df['trial_id'].isin(good_trial_ids)]
```

Missing:
- Comparison against training corpora
- N-gram overlap detection
- Semantic similarity computation
- Contamination severity scoring
- Mitigation recommendations

## Key Observations

### Repository Purpose Mismatch
This is a cooperative multi-agent RL environment, not an evaluation framework for LLMs or general ML systems. The Stage 2 criteria assume dataset preparation and evaluation infrastructure building, which are not the primary goals of this repository.

### Human Data Collection Focus
The most relevant "preparation" work is in human gameplay data collection and processing:
- Structured schema for game trajectories
- CSV to DataFrame conversion
- Basic trial filtering
- But no comprehensive evaluation preparation pipeline

### Deprecated Training Code
The README explicitly states:
```markdown
NOTE + LOOKING FOR CONTRIBUTORS: DRL and BC implementations are now deprecated.
```

This means even the limited training infrastructure is no longer maintained, further indicating this isn't an active evaluation framework.

### Demo-Focused Architecture
The `overcooked_demo` module prioritizes web-based human-AI interaction:
- Flask/SocketIO server for real-time gameplay
- Browser-based interface
- Game state visualization
- Not evaluation artifact preparation

## Conclusion

Overall Stage 2 Score: 3/24 (12.5%)

Overcooked-AI is a specialized game environment for multi-agent RL research, not a general evaluation framework. It provides minimal data preprocessing for human gameplay logs and basic model loading, but lacks the comprehensive dataset preparation, quality assessment, infrastructure building, and scenario generation capabilities expected from an evaluation harness. The repository's focus on cooperative gameplay simulation rather than evaluation preparation results in very low scores across all Stage 2 features.
